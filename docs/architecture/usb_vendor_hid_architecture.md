# USB Vendor HID Architecture

USB wired mode is the primary PC connection path. The product uses a composite
USB device with standard HID input plus a Vendor HID control channel.

## Interface Model

```text
USB Composite Device
  Interface 0: Standard HID Keyboard
  Interface 1: Consumer/System HID, if needed
  Interface 2: Vendor HID Control Channel
  Optional: CDC ACM debug channel, development builds only
```

Vendor HID is the product control path. CDC is not part of the product control
architecture.

## Responsibilities

```text
V3F:
  generate keyboard HID reports

V5F:
  own USB stack
  send HID reports to USB endpoints
  own Vendor HID control channel
  optionally expose CDC debug
```

V5F must not move keyboard behavior policy out of V3F just because the USB stack
runs on V5F. V5F is a transport sender for V3F-generated reports.

## Standard HID Keyboard

The keyboard interface must work without PC software.

Requirements:

- standard boot/protocol-compatible keyboard behavior where feasible
- normal typing without Local Core Service
- deterministic report ordering
- report overrun diagnostics
- no dependency on Vendor HID for basic input

## Consumer/System HID

Consumer/system HID can be added for:

- media keys
- system keys
- screen/brightness-related standard usages, if appropriate

Consumer/system reports may be generated by V3F if they are part of keyboard
behavior tables, then sent by V5F.

## Vendor HID Control Channel

Vendor HID carries product control messages:

- hello and capability negotiation
- Device Current Config read/write
- config change records
- screen state projection
- notification and Agent unavailable state
- diagnostics
- firmware update transfer commands
- slot mapping for software-projected entities

Vendor HID messages are device protocol frames. They are not UI WebSocket JSON.

## CDC Debug Channel

CDC may be kept for development:

- debug logs
- shell/diagnostics during bring-up
- emergency manual inspection

CDC requirements:

- compile-time enable/disable
- disabled in normal product builds unless explicitly needed
- not required by PC software for product features
- not used as the primary device protocol

## Report and Frame Boundaries

HID keyboard reports and Vendor HID control frames are separate concerns.

```text
Keyboard HID report:
  realtime input output
  generated by V3F
  sent by V5F

Vendor HID frame:
  control/config/status protocol
  handled by V5F
  may install config that later affects V3F
```

Control messages must not block keyboard report sending. Long operations such
as firmware update or config sync should be chunked and scheduled so keyboard
input remains responsive.

## Capability Negotiation

After PC software opens Vendor HID:

1. PC sends `HELLO`.
2. Device returns identity and protocol version.
3. PC requests capabilities.
4. Device returns supported message families and limits.

Minimum capability fields:

- hardware revision
- firmware version
- protocol version
- USB product/interface identifiers
- max Vendor HID payload size
- supported config schema version
- supported runtime table version
- firmware update support

## MVP Messages

Vendor HID MVP must support:

```text
HELLO
GET_CAPABILITIES
GET_CURRENT_CONFIG_SUMMARY
READ_CURRENT_CONFIG_CHUNK
WRITE_CURRENT_CONFIG_BEGIN
WRITE_CURRENT_CONFIG_CHUNK
WRITE_CURRENT_CONFIG_COMMIT
GET_CONFIG_REVISION
GET_DIAGNOSTICS_SUMMARY
ERROR
```

Keyboard HID typing can exist before the full control protocol, but the MVP is
complete only when current config read/write works through Vendor HID.

## Debug and Diagnostics

USB diagnostics should include:

- enumeration success/failure
- active interfaces
- HID report queue depth/overrun
- Vendor HID last command
- Vendor HID framing errors
- payload-too-large errors
- config write progress

## Implementation Note

The current repository already includes CherryUSB HID/CDC composite references.
Implementation should replace demo-oriented HID/CDC behavior with product-owned
Keyboard HID and Vendor HID modules while preserving CDC as debug-only.
